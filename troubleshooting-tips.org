* Troubleshooting tips
** 1. Prioritize Observability: Reduce System State Uncertainty
The single biggest time sink in debugging is a lack of information. Your first goal is to force the system to reveal its precise state at the point of failure.
*** Fix the Debugging Pipeline (Eliminate Silent Errors)
If an error is vague, non-specific (e.g., "An error occurred," "Skipping sample"), or completely suppressed, **assume your logging is the problem**, not the code itself.
**** Action: Immediately wrap the failing section in a handler that prints a **full, language-specific traceback** (e.g., `traceback.print_exc()` in Python).
**** Goal: Transform the error from a vague guess into a precise file and line number.
*** Validate Inputs at Every Boundary (The Data Contract)
Errors frequently occur at the hand-off points between separate components (e.g., a build tool writing a file for a script to read, a function preparing a dictionary for a framework).
**** Principle: Confirm that the output of component A (the provider) honors the exact expectations (the "contract") of component B (the consumer).
**** Action: Use debugging statements to inspect the **data structure's shape, content, and key names** immediately before and after every major transformation or I/O operation.
** 2. Decompose and Isolate (The Layered Approach)
Break down the system into the smallest testable units and isolate the failure to one layer or one function.
*** Test Components in Isolation (Surgical Debugging)
When a large system fails, the bug is often masked by a separate, innocuous feature that changes the system state.
**** Action: Temporarily comment out all *non-essential* functionality (e.g., network calls, database writes, UI updates) and run the simplest possible test case that still triggers the error.
**** Goal: Strip the failure down to a single function or line of code that can be reliably reproduced. This prevents a minor logic error from masquerade as a complex logic error deep inside the program.
** 3. Pinpoint the Nature of the Failure (The Error Taxonomy)
Successful troubleshooting depends on correctly classifying the error.
*** Distinguish Between Error Categories
Be ready to pivot your strategy based on the error type:
**** Logic Errors (Business Rules)
The code runs but produces the wrong result (e.g., filtering logic drops too many items). Requires stepping through the code's execution flow.
**** Data Integrity Errors (Input/Contract)
The code fails because the input data violates an assumption (e.g., a NumPy array is empty, a required dictionary key is missing, leading to `KeyError`). Requires focusing on input validation and boundary checking.
**** Configuration Errors (Framework Mismatch)
The code fails because an external library or framework requires a parameter, version, or specific data structure that wasn't provided (e.g., `NameError: PeptModel`). Requires checking documentation.
*** Consult External Documentation for Configuration Errors
If the error references a specific class, parameter, or constant used by a large framework (e.g., `group_by_length`, `input_ids`), treat it as a documentation problem.
**** Principle: Use the error message as a direct search query against the framework's official API documentation.
**** Goal: Quickly confirm if the argument you provided matches the framework's required syntax, constraints, or expected data format, bypassing internal code debugging.

** 5. Intermittency Usually Implies Contention
Bugs that happen "sometimes," or after "waiting a while," or depend on "heavy load" are rarely logic errors (like an off-by-one bug). They are almost always resource contention issues (race conditions). When time is a variable, look for two things fighting for the same resource.
** 6. Prove the Negative
Proving a negative is often the fastest way to eliminate entire categories of hypotheses.
*** Principle: Instead of trying to confirm what the bug *is*, prove what it *isn't*. A single test that rules out an entire class of causes (e.g., "the bug reproduces without the network, so it's not a network issue") is worth more than ten tests that incrementally narrow within a single class.
*** Action: For each hypothesis category, design the cheapest possible test that would conclusively rule it out. Run those elimination tests before pursuing any single hypothesis in depth.
** 7. Don't Debug the Library Before the System
Trust the library until you have verified that the system (OS/Kernel) is delivering the raw materials correctly.
** 8. The "Trap" Technique
When logs are ambiguous, create a "trap." Writing small, aggressive diagnostic traps is often more effective than passive logging.
** 9. Trace the Lifecycle, Not the Error Location
Don't focus on where the error is reported. Instead, find where the problematic value is *created*, *assigned*, and *used*. The root cause is usually upstream of where symptoms appear.
*** Action: Search for all assignment sites, not just the line the error mentions.
*** Goal: Understand the full journey of the data or object before deciding where to intervene.
** 10. Map All Possibilities Before Fixing
Before adding any fix, enumerate every value or state the problematic element can hold across all code paths. Write them down explicitly.
*** Principle: The correct fix often emerges naturally once you see the complete picture.
*** Anti-pattern: Fixing individual error locations without understanding the whole system leads to whack-a-mole debugging.
** 11. Fix at the Source, Not the Symptom
When multiple errors share a common cause, resist fixing each error individually. Look for the single upstream change that resolves all of them.
*** Principle: One declaration or structural fix beats N scattered guards or assertions.
*** Action: When you see errors on lines 50, 75, and 120, ask "What single change would make all three disappear?" rather than "How do I fix line 50?"
** 12. Look for Missing Declarations, Not Missing Guards
When a type checker or validator complains, the first question should be "Is this properly declared?" not "How do I guard this usage?"
*** Principle: Guards and assertions are downstream fixes; declarations and contracts are upstream fixes. Prefer upstream.
*** Action: Before adding runtime checks, verify that the data structure, type, or schema is correctly specified at its point of definition.
** 13. Verify Units, Scales, and Formats at Boundaries
When data crosses component boundaries, verify that producer and consumer agree on units, scales, and formats.
*** Principle: Many bugs are "impedance mismatches" - one component produces data in one format while another expects a different format. Examples: probabilities vs logits, UTC vs local time, 0-indexed vs 1-indexed, bytes vs strings.
*** Action: At integration points, explicitly document and verify the expected format on both sides.
** 14. When Filters Return Empty, Inspect the Rejection
If any filtering operation (threshold, query, predicate) returns zero results when you expect many, the data usually exists but fails the filter criteria.
*** Principle: "No results" from a filter rarely means "no data" - it usually means "data exists but doesn't match."
*** Action: Before investigating data sources, print the filter criteria alongside sample values being filtered to see why they're rejected.
** 15. Trace the Full Pipeline in One Pass
Before adding diagnostics incrementally, map out the complete data flow and instrument every stage simultaneously.
*** Principle: Incremental debugging (add one print, run, add another, run) wastes cycles. A single test run with visibility into all stages reveals the failing boundary immediately.
*** Action: Identify all transformation stages, add logging at each boundary in one edit, then run once and analyze the full trace.
** 16. Binary Search the Pipeline
When data flows through N stages and output is wrong, don't debug linearly from stage 1. Start at the middle stage.
*** Principle: If the middle stage shows correct data, the bug is downstream. If wrong, it's upstream. This finds the failing stage in log(N) steps instead of N.
*** Action: For a 6-stage pipeline, check stage 3 first, then narrow to either 1-3 or 4-6, then pinpoint.
** 17. Identify What Changed Between Working and Failing
When something worked before but fails now (or works in one environment but fails in another), explicitly list the differences.
*** Principle: The bug is almost always in the delta. Comparing "mock vs real," "dev vs prod," or "before vs after" focuses attention on the actual change.
*** Action: Write down: "What is different?" Then verify each difference until you find the one that matters.
** 18. Read the Integration Code Before Adding Diagnostics
Before instrumenting code with debug statements, read the source of the components involved - especially at integration points.
*** Principle: Five minutes reading often reveals assumptions, edge cases, or transformations that explain the bug without any debugging.
*** Action: When component A calls component B and the result is wrong, read B's implementation first. Check what it actually returns, not what you assume it returns.
** 19. Write Down and Verify Assumptions Explicitly
When debugging, you hold implicit assumptions about data formats, value ranges, and component behavior. These assumptions are often wrong.
*** Principle: Unexamined assumptions are the primary source of wasted debugging time. Making them explicit forces verification.
*** Action: Before deep-diving, list your assumptions as checkboxes. Verify each one with evidence before proceeding.
** 20. When "No Response," Ask About Timing First
If a system appears unresponsive, ask when (not just what) output appears. Buffering issues are invisible if you only examine content.
*** Principle: "No response" often means "delayed response due to buffering." Output that appears only after process exit, window close, or timeout indicates a pipe, stream, or I/O buffering problem - not a logic error.
*** Action: When something "doesn't respond," immediately ask: "Does output appear instantly, or only after closing/exiting?" If delayed, investigate buffering at I/O boundaries (pipes, sockets, streams) before debugging application logic.
** 21. Get a Stack Trace Before Forming Any Hypothesis
When a function fails unexpectedly, enable the language's debugger or traceback facility and reproduce the failure before theorizing about the cause.
*** Principle: A stack trace tells you exactly where the failure is. Without one, you are guessing which layer is broken, and each guess costs a full investigate-test cycle. Even experienced developers routinely guess wrong.
*** Action: Before forming any hypothesis, enable full error reporting (e.g., debugger break-on-error, verbose tracebacks, crash dumps) and trigger the failure once. Read the trace top-to-bottom. Only then decide what to investigate.
*** Anti-pattern: Theorizing about the cause, implementing a speculative fix, testing it, and discovering it didn't help — then repeating this loop multiple times before finally looking at the actual error.
** 22. Check User Customization Before Blaming the System
When a well-established feature or library function fails, user-injected code is the most likely culprit — not a bug in the library or the system.
*** Principle: Hooks, callbacks, plugins, middleware, decorators, monkey patches, interceptors, and configuration files all inject user code into standard code paths. Any of these can crash, alter control flow, or corrupt state in ways that make the standard feature appear broken.
*** Action: Before diving into library internals, enumerate all user customization points that run during the failing operation (event hooks, lifecycle callbacks, registered plugins, advice/aspects, middleware chains). Disable them and test. If the feature works without them, re-enable one at a time to find the culprit.
** 23. Verify Your Hypothesis Explains the Full Failure
Before implementing a fix, ask: "Does this hypothesis fully account for every observed symptom?" If it doesn't, you are fixing a symptom, not the cause.
*** Principle: A correct root-cause hypothesis must explain all the evidence. If your theory explains why output looks wrong but not why a function returns zero results, the theory is incomplete — and any fix based on it will not resolve the real problem.
*** Action: Before writing any fix, list every observed symptom. Check each one against your hypothesis. If any symptom remains unexplained, keep investigating instead of implementing a partial fix.
*** Anti-pattern: Observing that component A has a plausible defect, fixing it, and discovering the real failure was in component B — which you never examined because the defect in A seemed like a sufficient explanation.
